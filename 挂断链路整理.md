# AOSP 挂断链路：从界面/Modem 到挂断通知下发

> Android Telephony 挂断流程，排查"挂不断"必备

---

## 一、整体架构

挂断分 **两条方向**：

```
下行挂断（用户主动挂）：  InCallUI → Telecom → Telephony → RIL → Modem
上行挂断（对方/网络挂）： Modem → RIL → Telephony → Telecom → InCallUI
```

## 一.5 Telephony vs Telecom 在挂断中的角色

链路中的 Telecom 和 Telephony 是两个独立层级，职责不同：

| | Telephony | Telecom |
|--|--|--|
| **职责** | 管硬件，跟基站通信（发 ATH 命令） | 管电话，管理通话状态机（ACTIVE → DISCONNECTED） |
| **下行挂断中** | 执行 `hangup()` → RIL → `RIL_REQUEST_HANGUP` → AT 命令 | 接收 UI 请求，更新状态，通知 ConnectionService 执行断开 |
| **上行挂断中** | 收到 Modem `NO CARRIER` → CallTracker 轮询确认 | 接收状态变化 → 更新 Call 为 DISCONNECTED → 通知 InCallUI |
| **桥梁** | **`ConnectionService`** 把 Telephony 的底层通话对象包装成 Telecom 的统一通话对象 |

```
用户挂断:
  InCallUI → Telecom（状态更新） → ConnectionService（桥梁） → Telephony（发 ATH）

对方挂断:
  Modem（NO CARRIER） → Telephony（轮询确认） → ConnectionService（桥梁） → Telecom（通知 UI）
```

**Call 对象在挂断流程中的角色变化：**

```
用户挂断：
  Telecom 的 Call（ACTIVE） → 发起 disconnect → 状态 ACTIVE → DISCONNECTING → DISCONNECTED
  Telephony 内部 Call 同步更新 → 发 ATH 给 Modem → Modem 返回 OK

对方挂断：
  Modem 上报 NO CARRIER → Telephony 内部 Call 状态更新
  → ConnectionService 同步到 Telecom 的 Call → 状态 ACTIVE → DISCONNECTED → InCallUI 关闭界面
```

## 二、用户主动挂断（完整下行链路）

```
InCallActivity — 用户点击挂断按钮
  ↓
InCallPresenter（拨号器内部事件分发，具体类名因实现而异） → InCallService.disconnect(call)
  ↓
InCallService → TelecomService → CallsManager.disconnect()
  ↓
CallsManager 更新状态：ACTIVE → DISCONNECTING
  ↓
ConnectionService.disconnect()
  ↓
TelephonyConnectionService → disconnect()
  ↓
Call.hangup() → GsmCdmaPhone.hangup()
  ↓
GsmCdmaCallTracker.hangup(call)
  ↓
RIL.java → mCommands.hangup(connection.hangupString, ...)
  ↓
RIL_HANGUP() → 通过 socket 发送消息给 RILD
  ↓
RILD 收到 RIL_REQUEST_HANGUP → 解析 hangupString
  ↓
发 AT 命令到 Modem：ATH 或 AT+CHLD
  ↓
Modem 执行挂断 → 返回 OK
```

## 三、对方主动挂断（上行链路）

```
Modem 检测到对方挂断 → AT 通道上报：
  NO CARRIER（通话断开）
  +CAUSE: 16（Q.850 原因码，部分 Modem 支持）
  ↓
RILD 解析 → RIL_onUnsolicitedResponse()
  ↓
RIL.java 收到 socket 消息 → unsolResponse()
  → handleCallStateChanged()
  → 发送 UNSOL_RESPONSE_CALL_STATE_CHANGED
  ↓
CallTracker.pollCallsWhenSafe() → AT+CLCC 查询
  ↓
handlePollCalls() → 发现通话已消失/状态变化
  ↓ 计算挂断原因 → DisconnectCause
  ↓
Phone.notifyPhoneStateChanged()
  ↓
TelecomAdapter → TelecomService
  ↓
CallsManager 更新 Call 状态：ACTIVE → DISCONNECTED
  ↓
ConnectionService → InCallService.onCallRemoved(call)
  ↓
InCallActivity 关闭通话界面
```

## 四、关键时序对比

```
用户主动挂断：
InCallUI → Telecom → ConnectionService → CallTracker → RIL → Modem

对方主动挂断：
Modem → RIL → CallTracker → Telecom → InCallUI
```

## 五、关键类速查

| 类 | 层级 | 挂断中的职责 |
|----|------|-------------|
| `InCallPresenter` | InCallUI | 接收用户挂断点击，通知 InCallService |
| `InCallService` | android.telecom (AOSP 框架 API) | 桥接 UI 和 Telecom 的桥梁，系统回调接口 |
| `CallsManager` | framework/telecom | 管理通话状态，调用 ConnectionService.disconnect |
| `ConnectionService` | framework/telecom | 抽象连接服务，Telecom 到 Telephony 的桥梁 |
| `TelephonyConnectionService` | framework/telephony | 具体实现，调用 Phone 层挂断 |
| `GsmCdmaCallTracker` | framework/telephony | 执行挂断逻辑，调用 RIL |
| `RIL.java` | framework/telephony | 向 RILD 发送 RIL_REQUEST_HANGUP |
| `rild` | vendor 层 | 收到 RIL 消息，发 AT 命令给 Modem |

## 六、hangup 是如何下发到 Modem 的（最后一公里）

**一句话**：`at_send_command("ATH")` 这一行就是 AT 命令真正通过串口/USB 写到 Modem 的那一行。

### 完整路径

```
GsmCdmaCallTracker.hangup()
  ↓ 调用 RIL.java 的 mCi.hangup()
RIL.java.hangup()
  ↓ 封装 RIL_REQUEST_HANGUP
  ↓ 通过 socket 发送
RILD 收到 RIL_REQUEST_HANGUP
  ↓ 调用 onRequest(RIL_REQUEST_HANGUP, ...)
vendor ril（C++ 库，厂商自定义实现）
  ↓ 打开 /dev/ttyUSBx（AT 通道）
  ↓ 写入字符串: "ATH\r" 或 "AT+CHLD=x\r"
Modem 收到 AT 命令
```

### AT 命令写下去的那一行

在 vendor ril 的 C 代码里：

```c
// vendor ril 的 C 层实现（厂商自定义）
at_send_command("ATH", NULL);
```

底层会通过串口（tty）或 USB 通道把字符串写到 Modem。再往下就是硬件层通信，Android 软件层到这就结束了。

## 七、排查思路（"为什么没下发挂断通知给 Modem"）

### 7.1 挂断请求根本没发出来

**排查方向**：从 InCallUI 开始逐层检查

| 层级 | 查什么 | 日志关键词 |
|------|--------|-----------|
| InCallUI | 挂断按钮点击事件是否触发 | `InCallPresenter` `onDisconnectClicked` |
| InCallService | onDisconnect 是否被调用 | `InCallService` `disconnect` |
| Telecom | CallsManager 是否收到 disconnect 请求 | `CallsManager` `disconnect` |
| ConnectionService | disconnect 方法是否执行 | `TelephonyConnectionService` `onDisconnect` |
| CallTracker | hangup 是否被调用 | `GsmCdmaCallTracker` `hangup` |

### 7.2 RIL 层没发出去

**排查方向**：RIL.java → RILD

| 现象 | 原因 | 日志 |
|------|------|------|
| RIL 日志无 `RIL_REQUEST_HANGUP` | 挂断请求没到 RIL 层 | logcat 搜 `RIL_REQUEST_HANGUP` |
| RIL 报了 `RIL_E_INVALID_ARGUMENTS` | 参数错误（hangupString 为空） | 检查 call 对象的 hangupString |
| socket 发送超时 | RILD 进程无响应 | 看 RILD 进程是否存活 |

### 7.3 RILD 没发给 Modem

**排查方向**：RILD → Modem

| 现象 | 原因 | 日志 |
|------|------|------|
| RILD 日志无 `ATH` 或 `AT+CHLD` | RILD 没收到或没解析 | RILD 层日志 |
| Modem 返回 `ERROR` | Modem 状态异常（如通话已断开） | 看 AT 通道返回 |
| AT 通道超时 | Modem 无响应 | 看 RILD 超时日志 |

## 八、常见场景分析

### 场景一：用户点挂断，对方没挂断

```
排查步骤：
1. InCallUI 是否收到点击事件？ → 看 InCallPresenter 日志
2. Telecom 是否发起 disconnect？ → 看 CallsManager 日志
3. RIL 是否发出 RIL_REQUEST_HANGUP？ → 看 RIL 日志
4. Modem 是否返回 OK？ → 看 RILD/Modem 日志
```

**大概率断在**：RIL 层（socket 通信异常）或 Modem 层（AT 通道超时）

### 场景二：挂断后通话界面不关

```
排查步骤：
1. Modem 是否返回了挂断成功？ → 看 RIL 日志
2. CallTracker 是否收到状态变化？ → 看 handlePollCalls
3. Telecom 是否更新状态为 DISCONNECTED？ → 看 CallsManager
4. InCallService 是否收到 onCallRemoved？ → 看 InCallService
```

**大概率断在**：Telecom 状态机未更新，或 InCallUI 未绑定到 InCallService

### 场景三：挂断慢（用户点到对方挂断 > 2 秒）

```
排查步骤：
1. UI 响应延迟？ → 看 InCallUI 日志
2. RIL 请求排队？ → 看 RIL 日志中的请求时间戳
3. Modem 响应慢？ → 看 AT 命令从发送到返回的时间
4. 网络侧延迟？ → 看 Modem 侧的网络信令日志（需要 rillog）
```

## 九、关键日志标识

| 阶段 | RIL 日志 | Telecom 日志 |
|------|---------|-------------|
| 发起挂断 | `RIL_REQUEST_HANGUP` | `disconnect()` |
| 挂断请求到达 RILD | `RIL_REQUEST_HANGUP` received | — |
| Modem 返回 OK | `ATH` → `OK` | — |
| 断开上报 | `NO CARRIER` / `RING` 停止 | `SET_DISCONNECTED` |
| 对方挂断 | Q.850 原因码 16（Normal Call Clearing） | `DISCONNECT_REASON_REMOTE` |
| 自己挂断 | `ATH` | `DISCONNECT_REASON_LOCAL` |
| 挂断失败 | `RIL_E_GENERIC_FAILURE` | `disconnect failed` |

## 十、AT 命令速查

| 命令 | 用途 | 返回 |
|------|------|------|
| `ATH` | 挂断当前通话 | OK / ERROR |
| `AT+CHLD=1` | 挂断指定通话（多方通话） | OK / ERROR |
| `AT+CLCC` | 查询当前通话状态（挂断后验证） | +CLCC: ... |
| `AT+CPAS` | 查询手机活动状态 | +CPAS: 0(就绪) / 3(通话中) |

## 十一、面试高频考点

### Q1：用户挂断和对方挂断的链路有什么区别？

方向相反。用户挂断是自上而下的命令下发（UI → Telecom → Telephony → RIL → Modem），对方挂断是自下而上的状态上报（Modem → RIL → Telephony → Telecom → UI）。对方挂断的触发来源包括 AT 通道上报（如 NO CARRIER）和网络侧信令（SIP BYE / ISUP REL 等，取决于网络类型）。

### Q2：RIL 发送挂断请求后，Modem 返回 ERROR 怎么办？

CallTracker 会重试或上报挂断失败。上层收到 `RIL_E_GENERIC_FAILURE` 后，可能会重新查询通话状态（AT+CLCC）确认实际状态，避免状态不一致。

### Q3：为什么挂断后还要再发 AT+CLCC 查询？

因为 Modem 可能已经挂断但没正确上报，或者网络侧已经断开但 Modem 状态未更新。主动查询确保 Phone 对象的状态和 Modem 保持一致。
