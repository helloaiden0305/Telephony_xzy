## 一、VoLTE 基础概念

### 1.1 什么是 VoLTE

**VoLTE**（Voice over LTE），全称 Voice over Long Term Evolution，译名为**长期演进语音承载**，也被称为 **4G 高清通话**。

它是一种基于 4G LTE 网络的数据传输技术，主要用于提供高质量的语音通话服务。

---

### 1.2 VoLTE 的主要特点

| 特点 | 说明 |
|------|------|
| **高清语音** | 相比传统的 2G/3G 语音通话，音质更高、更清晰 |
| **通话与上网并行** | 在进行 VoLTE 通话时，用户可以同时进行 4G 网络上网 |
| **更快的接通速度** | VoLTE 技术的通话接通速度通常比传统通话更快 |

---

### 1.3 VoLTE 的核心原理

#### (1) 全 IP 化

```
传统 2G/3G：
通话建立 → 建立独占物理链路 → 通话结束才释放

VoLTE：
语音数据 → 分解成小数据包 → 通过 4G LTE 数据网络传输 → 与上网数据共享通道
```

#### (2) IMS（IP 多媒体子系统）

- IMS 提供了丰富的多媒体业务能力，使得 VoLTE 不仅能提供高质量的语音通话，还为未来的视频通话、媒体消息等业务奠定基础
- 通过 IMS，VoLTE 实现了会话控制，确保语音通话的稳定性和可靠性

#### (3) 高分辨率编解码技术

- VoLTE 采用先进的高分辨率编解码技术，对语音信号进行更高效、更高质量的编码和解码
- 语音传输的数据量更小，同时能保留更多的语音细节

#### (4) 数据与语音业务统一承载

```
CSFB 时代（4G 前）：
打电话 → 回落到 2G/3G 网络 → 无法同时上网

VoLTE 时代：
打电话 → 保持在 4G 网络 → 同时可以上网
```

#### (5) 更低的接入时延和掉线率

- 省去了回落到 2G/3G 网络的切换过程
- 4G 网络的稳定性和 QoS（服务质量）保障机制，使掉线率接近于零

---

### 1.4 总结

```
VoLTE = 4G LTE 高带宽 + IMS 控制能力 + IP 分组交换

效果：高清语音 + 通话上网并行 + 快速接通 + 低掉线率
```

## 二、VoLTE、IMS、LTE 的关联

### 2.1 一句话总结

```
LTE 是网络基础 → IMS 是业务平台 → VoLTE 是 IMS 上跑的语音业务
```

| 层级 | 作用 | 类比 |
|------|------|------|
| **LTE** | 提供高速数据通道 | 高速公路 |
| **IMS** | 提供语音服务架构 | 交通指挥中心 |
| **VoLTE** | 用 IMS 的语音能力在 LTE 上打电话 | 跑在高速上的语音服务 |

---

### 2.2 详细解释

#### LTE（4G 接入网络）

LTE 本质上提供的是：

- 数据承载（全 IP）
- 高速上网
- 低时延的数据链路

**但不提供 CS 语音（传统电话）功能。**

这就是为什么在 4G 网络下想打电话，需要一种"能在数据通道上跑语音"的服务。

#### IMS（IP Multimedia Subsystem）

IMS 是运营商构建的**全 IP 多媒体业务平台**。

IMS 承载的业务包括：

| 业务 | 说明 |
|------|------|
| **VoLTE** | 语音 |
| **VoWiFi** | WiFi 语音 |
| **RCS** | 增强短信 |
| **视频通话** | 基于 IMS |

```
IMS = 运营商的语音"服务器"和业务"控制平台"

手机只要注册上 IMS，就能使用上面所有业务能力（前提是运营商给你开通）。
```

#### VoLTE（Voice over LTE）

**VoLTE 是 IMS 上的语音业务。**

VoLTE 是 IMS 的一个"业务实例"，不是单独的技术。

## 三、IMS 的主要组成部分

### 3.1 核心组件

| 组件 | 全称 | 作用 | 类比 |
|------|------|------|------|
| **CSCF** | Call Session Control Function | 处理所有会话控制信令 | IMS 的"大脑" |
| **HSS** | Home Subscriber Server | 存储用户订阅信息、认证数据、路由信息 | 用户数据库 |
| **MGW** | Media Gateway | 媒体流转换（VoLTE 与传统电路交换网络互通时） | 协议转换器 |
| **AS** | Application Server | 提供各种增值业务和应用 | 业务服务器 |

### 3.2 CSCF 子模块

| 子模块 | 全称 | 作用 |
|--------|------|------|
| **P-CSCF** | Proxy CSCF | 代理 CSCF，第一入口点 |
| **I-CSCF** | Interrogating CSCF | 查询 CSCF，用户注册时查询 HSS |
| **S-CSCF** | Serving CSCF | 服务 CSCF，实际处理会话逻辑 |

## 四、VoLTE 和 5G VoNR

### 4.1 关系对比

| 技术 | 网络 | 说明 |
|------|------|------|
| **VoLTE** | 4G LTE | 4G 时代的 IP 语音解决方案 |
| **VoNR** | 5G NR | 5G 时代的 IP 语音解决方案 |

### 4.2 技术演进

```
2G/3G CS 语音 ──► VoLTE ──► VoNR
                     │            │
                     ▼            ▼
                  4G LTE      5G NR
                  (数据通道)    (数据通道)
                     │            │
                     └─────┬─────┘
                           │
                           ▼
                        IMS 核心网
                    (语音控制平台)
```

### 4.3 5G 部署早期的特殊情况

```
┌─────────────────────────────────────────────────────────────────┐
│  非独立组网（NSA）5G 模式下：                                     │
│                                                                 │
│  用户连接 5G 信号 ──► 但语音通话走 4G VoLTE                      │
│                                                                 │
│  原因：5G 核心网尚未完全成熟或部署                                 │
│                                                                 │
│  5G 终端必须根据当前网络能力决定：                                 │
│  • 语音承载走 VoLTE（LTE）                                        │
│  • 或走 VoNR（NR）                                               │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 总结

```
• VoLTE 是 4G 网络的 IP 语音技术
• VoNR 是 5G 网络的 IP 语音技术，是 VoLTE 的演进升级
• 在 5G 部署早期，VoLTE 为 5G 语音提供过渡和兼容性支持
• 两者都基于 IP 分组交换原理，依赖 IMS 核心网提供服务
```

## 五、实机中 VoLTE 属性

### 5.1 两大组成部分

```
VoLTE 正常工作 = 设备内部控制 + 运营商外部配置
```

| 维度 | 说明 |
|------|------|
| **设备内部控制** | 手机系统层面的开关和配置 |
| **运营商外部配置** | 网络是否支持 + 用户是否开通 |

---

### 5.2 设备内部属性：msc.config.volte_on

**msc.config.volte_on** 是 VoLTE 功能的**总开关**。

#### 作用范围

| 模块 | 影响 |
|------|------|
| **TelephonyPropertiesEx.java** | 判断 VoLTE 功能是否被系统支持或开启 |
| **IMS 服务** | 直接控制 IMS 服务是否启动 VoLTE 注册、处理呼叫 |
| **多卡管理** | 判断多 SIM 卡场景下 VoLTE 功能是否可用 |
| **Telecom 框架** | 影响 Telecom 框架如何处理 VoLTE 呼叫 |
| **用户界面** | 控制设置界面是否显示 VoLTE 相关选项 |

#### 属性来源

```
msc.config.volte_on 不是硬编码的，而是：
• 运行时动态构建
• 编译后期/刷机时/固件注入
• 根据"片区"信息（ro.product.locale.region 或 ro.boot.country）
```

---

### 5.3 运营商外部配置

| 属性 | 说明 | 获取方式 |
|------|------|---------|
| **carrier_volte_available_bool** | 运营商网络是否提供 VoLTE 服务 | dumpsys carrier_config |
| **carrier_volte_provisioned_bool** | 用户 SIM 卡/套餐是否已开通 VoLTE | dumpsys carrier_config |

## 六、APN 配置

### 6.1 什么是 APN

**APN（Access Point Name）** 是终端接入运营商数据网络的**入口配置**。

```
APN = 手机连接运营商网络的"专属通道"

不同业务 → 需要走不同的通道
```

### 6.2 核心 APN 类型

| APN 类型 | 用途 | 类比 |
|---------|------|------|
| **Default APN** | 上网通道（微信、视频、游戏等） | 社会车辆主干道 |
| **IMS APN** | 高清通话专属通道 | 救护车/公交专用道 |
| **MMS APN** | 彩信业务 | 快递专用通道 |

---

### 6.3 APN 加载机制

```
┌─────────────────────────────────────────────────────────────────┐
│  SIM 卡决定用哪套 APN（随卡配置）                               │
│                                                                 │
│  SIM 卡中存储：MCC（国家码）+ MNC（运营商码）                   │
│                              │                                 │
│                              ▼                                 │
│              根据 MCC+MNC 自动加载对应配置：                     │

│                                                                 │

│  • 插中国移动卡 → 加载 46000 的 APN 配置                      │
│  • 插中国联通卡 → 加载 46001 的 APN 配置                      │
```

### 6.4 IAAPN（Initial Attach APN）

#### 定义

IAAPN 是设备**首次连接到 LTE 网络时使用的关键配置**，用于建立 LTE 数据连接的"第一条命脉"。

```
IAAPN 使用场景：
• LTE 首次 Attach 阶段
• Attach 成功后，自动切换到正常 APN
```

#### 作用

```
┌─────────────────────────────────────────────────────────────────┐
│                        IAAPN 的重要性                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  IAAPN 异常 → LTE Attach 失败                                   │
│       │                                                         │
│       ▼                                                         │
│  LTE Attach 失败 → LTE 注册失败                                  │
│       │                                                         │
│       ▼                                                         │
│  LTE 注册失败 → IMS 无法注册                                      │
│       │                                                         │
│       ▼                                                         │
│  IMS 无法注册 → VoLTE 不能工作                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### MTK 平台特殊配置

```
问题：Labu 海外 4G LTE 无法注册（APN 配置缺失）

原因：
• 很多平台（含部分 MTK 基线）会自动补一个默认 IAAPN
• 但升级基线中，自动补全逻辑可能被移除或不再生效

解决：在以下路径添加 IAAPN 配置：
/cust/global/carrier/(mccmnc)/xml/apns-conf.xml

示例配置：
<apn carrier="IA"
    mcc="001"
    mnc="01"
    apn=""
    type="ia"
    protocol="IPV4V6"
    roaming_protocol="IPV4V6"
/ >
```

### 6.5 Default APN vs IMS APN

| 对比维度 | Default APN | IMS APN |
|---------|------------|---------|
| **用途** | 上网（微信、视频、游戏等） | 打电话（VoLTE）和发短信 |
| **状态栏表现** | 有 4G/5G 数据上下行箭头 | 无 |
| **异常表现** | 提示"无网络连接" | 通话功能异常 |

---

### 6.6 常见问题：切卡后无法上网

#### 现象

```
用户切换回实体卡后：
• 手机有信号 ✓
• 可能能打电话 ✓
• 但无法上网（没有移动数据）✗
```

#### 根因

```
底层真实情况：
系统在切换卡之后，只帮用户建好了"打电话的通道"（IMS APN），
却忘记/漏掉了去建立"上网的通道"（Default APN）

"上网通道"根本没有被系统发起请求去建立，所以实体卡没有移动数据。
```

#### 通俗比喻

```
运营商网络 = 高速公路

Default APN = "社会车辆主干道"（给所有 App 上网用的）
IMS APN = "救护车/公交专用道"（只给打电话用的）

问题：切换卡后，只打开了专用道的闸门，主干道的闸门死死关着
结果：路通了（有信号），但车开不过去（数据业务无法使用）
```

#### 研发排查方向

```
重点检查：
为什么切卡之后，系统只去开 IMS 的闸门，
而不去开 Default（上网）的闸门？

是不是切卡的时候状态卡住了，导致系统以为这张卡不需要上网？
```

## 七、VoIP 与 VoLTE 的关系

### 7.1 VoIP 基础

**VoIP（Voice over IP）** 是一类统称概念，指任何基于 IP 网络传输语音的技术或应用。

```
VoIP 核心特征：
• 语音被编码为 IP 数据包
• 通过互联网或私有 IP 网络传输
• 不依赖传统电路交换（CS）语音网络

从技术定义上看：VoLTE / VoNR 本质上也是 VoIP 的一种专业化形态
```

### 7.2 VoIP 的典型形态

| 类型 | 说明 | 示例 |
|------|------|------|
| **OTT VoIP** | 互联网应用级，运行在应用层，基于公网传输 | 微信语音、WhatsApp |
| **企业/私有 VoIP** | 运行在企业内网或专网 | 企业办公电话系统 |
| **运营商级 IP 语音** | 运营商 LTE/NR 网络，由 IMS 控制 | VoLTE、VoNR |

---

### 7.3 VoIP 与 VoLTE 的本质区别

| 对比维度 | 普通 VoIP | VoLTE / VoNR |
|---------|----------|-------------|
| **网络层级** | 公网 IP 网络或普通数据链路 | 运营商 LTE/NR 专用承载网络 |
| **IMS 依赖** | 不依赖，由应用或私有服务器控制 | 强依赖 IMS |
| **QoS 保障** | 与普通数据混传，拥塞时优先级不可控 | 专用 EPS Bearer，端到端 QoS |
| **号码体系** | 可能使用账号 ID，与传统电话互通能力有限 | 标准手机号码，可与 PSTN 无差别互通 |

---

### 7.4 终端视角的区别

```
┌─────────────────────────────────────────────────────────────────┐
│                        VoLTE（终端视角）                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  • 深度集成在 Telephony Framework 中                            │
│  • 由 IMS Service 管理                                          │
│  • 与 RIL / Modem / Carrier Config 紧密联动                    │
│                                                                 │
│  关闭 msc.config.volte_on → IMS 不启动 → VoLTE 无法使用         │
│  但并不影响任何 VoIP 应用的使用                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                      普通 VoIP（终端视角）                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  • 运行在应用沙箱                                                │
│  • 仅使用系统提供的数据网络                                      │
│  • 不参与 Telephony 语音链路                                    │
│                                                                 │
│  关闭 msc.config.volte_on → VoIP 应用不受影响                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 7.5 VoIP / VoLTE / VoNR 关系总结

```
包含关系：
┌─────────────────────────────────────────────┐
│              VoIP（IP 语音统称）              │
│  ┌─────────────────────────────────────┐   │
│  │  VoLTE（LTE 网络上的 VoIP）         │   │
│  └─────────────────────────────────────┘   │
│  ┌─────────────────────────────────────┐   │
│  │  VoNR（5G NR 网络上的 VoIP）        │   │
│  └─────────────────────────────────────┘   │
└─────────────────────────────────────────────┘

本质关系：
VoLTE / VoNR = 电信级 + 有 IMS + 有 QoS 的 VoIP
```

### 7.6 为什么运营商必须采用 VoLTE

| 原因 | 说明 |
|------|------|
| **电信级可靠性** | 需要与传统电话网络同等的稳定性 |
| **无缝互通** | 需要与 PSTN/CS 网络无差别互通 |
| **统一管理** | 计费、号码、服务质量统一管控 |

```
普通 VoIP 无法满足上述任何一项。
```
