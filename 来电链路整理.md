# AOSP 来电链路：从 Modem 到界面显示

> Android Telephony 核心流程，面试必备

---

## 一、整体架构（五层模型）

```
┌─────────────────────────────────────────────┐
│  InCallUI (App 层)                           │  ← 用户界面
│  InCallActivity / InCallService              │
├─────────────────────────────────────────────┤
│  Telecom Service (System Server)             │  ← 通话管理中心
│  CallsManager / Call / ConnectionService     │
├─────────────────────────────────────────────┤
│  Telephony Framework (Telephony Framework)   │  ← Phone 对象管理
│  GsmCdmaPhone / CallTracker / RIL.java       │
├─────────────────────────────────────────────┤
│  RIL Daemon (vendor 层, C++)                │  ← AT 命令解析
│  rild / vendor ril（厂商自定义 C++ 库）         │
├─────────────────────────────────────────────┤
│  Modem / Baseband                            │  ← 射频/基带
│  AT 命令: RING / +CLIP                       │
└─────────────────────────────────────────────┘
```

**关键分层原则**：
- **Modem**：基带芯片，负责无线信号
- **RIL Daemon + RILJ**：中间桥梁，翻译 AT 命令 ↔ Java 对象
- **Telecom**：Android 6.0 (Marshmallow) 引入的统一通话管理服务，是 framework 层核心
- **TelephonyRegistry**：Android 13+ 集中管理 Telephony 状态回调，部分原来由 CallTracker 直接通知上层的逻辑改走 TelephonyRegistry
- **InCallUI**：纯 UI 层，不处理通话逻辑

## 一.5 Telephony vs Telecom 核心区分（面试高频）

```
Telephony 层                          Telecom 层
───────────                          ──────────
管硬件（跟基站说话）                    管电话（通话管理、路由、多路切换）
核心类: RIL, Phone, CallTracker        核心类: CallsManager, android.telecom.Call
"基站说有个电话来了"                     "好，我来管理这个通话，通知 UI 响铃"
```

**真正的桥梁是 `ConnectionService`** —— 它负责把 Telephony 的内部 Call 包装成 Telecom 对外暴露的 Call：

```
Modem 上报来电
  → RIL.java（搬运工：翻译 AT 命令）
    → CallTracker 创建 GsmCdmaCall（Telephony 内部模型，写满底层信息）
      → ConnectionService 包装成 android.telecom.Call（对外统一接口）
        → Telecom 接管 → 通知 InCallUI 显示
```

**注意**：Telephony 和 Telecom 各有一个 Call 对象，不是同一个东西。ConnectionService 就是分拣中心，把内部单转成客户单。

**Call 对象在来电流程中的角色变化：**

```
阶段 1：不存在              →  Modem 上报 RING + CLIP
阶段 2：Telephony 内部 Call  →  CallTracker 创建 GsmCdmaCall
                              承载：底层信息（AT+CLCC 结果、通话 ID、号码、状态）
阶段 3：Telecom 对外 Call    →  ConnectionService 包装成 android.telecom.Call
                              承载：统一接口（号码、状态、接听/挂断操作）
阶段 4：UI 展示的 Call       →  InCallService 收到回调，UI 直接绑定到 Telecom 的 Call
```

## 二、核心流程（6 步）

### Step 1：Modem AT 上报

```
Modem 检测到网络侧来电 → 通过 AT 通道上报：

  RING
  +CLIP: "13800138000",128,"",0,"",1
  
RING        → 来电振铃指示
+CLIP       → 来电号码显示（Calling Line Identification Presentation）
  "138..."   → 来电号码
  128        → 号码类型
```

### Step 2：RIL Daemon 接收并上报

```
vendor ril（厂商自定义 C++ 库）解析 AT 命令：
  ↓
  收到 "RING" → 调用 RIL_onUnsolicitedResponse()
  ↓
  通过 socket 发送给 framework 层的 RIL.java
```

**Unsolicited Response**：这是 Modem 主动上报的消息，不需要 framework 主动请求。

### Step 3：RIL.java 处理并广播

```
RIL.java 收到 socket 消息 → unsolResponse() 分发：

  RING → handleCallStateChanged()
       → 触发 mCallStateRegistrants 通知所有监听者
       → 发送 UNSOL_RESPONSE_CALL_STATE_CHANGED

+CLIP → 解析出号码，存入 Call 对象
```

### Step 4：CallTracker 轮询通话状态

```
收到状态变化通知后，CallTracker.pollCallsWhenSafe()：

  ↓ 主动向 Modem 查询当前通话状态（AT+CLCC）
  ↓ 收到完整通话列表
  ↓ handlePollCalls() 对比前后状态差异
  ↓ 发现有新来电（NEW → INCOMING）
  ↓ Phone.notifyPhoneStateChanged() 广播状态变化
```

**关键设计**：不是纯被动等待 Modem 推送，而是**收到通知后主动轮询**。这样确保拿到完整状态。

### Step 5：Telecom 管理通话

```
Telephony 通过 TelecomAdapter 通知 Telecom：

  ↓ Telecom 处理来电意图（内部通过 CallsManager 路由）
  ↓ 创建 IncomingCall 对象
  ↓ Call 状态：NEW → RINGING
  ↓ 通知 InCallService.onCallAdded(call)
```

Telecom 的核心职责：
- 管理通话状态机（NEW / RINGING / ACTIVE / DISCONNECTED）
- 处理多路通话（通话中又来电话、三方通话等）
- 音频路由管理（听筒 / 蓝牙 / 免提）
- DTMF 按键音

### Step 6：InCallUI 显示

```
InCallService.onCallAdded(call) 触发：

  ↓ InCallPresenter（拨号器内部事件分发，具体类名因实现而异）
  ↓ 启动 InCallActivity（如果还没启动）
  ↓ 来电卡片（CallCard / CallCardFragment，UI 组件名因实现而异）渲染来电号码、接听/挂断按钮
  ↓ 同时触发来电铃声 + 震动
```

InCallUI 不负责通话逻辑，只负责**展示**和**用户交互**。

## 五、常见问题排查思路

| 现象 | 排查方向 | 关键日志 |
|------|---------|---------|
| **来电没铃声** | InCallUI 音频路由 | Telecom 日志，看 `onCallAdded` 是否触发 |
| **来电没显示** | InCallActivity 未绑定 | logcat 搜 `InCallService` `onCallAdded` |
| **来电无号码** | CLIP 未上报 | RIL 日志看 `+CLIP` 是否收到 |
| **来电挂断慢** | Telecom disconnect 逻辑 | Telecom 日志看 `SET_DISCONNECTED` |
| **通话中无法接听新来电** | Telecom 多路通话逻辑 | `CallsManager` 看来电路由日志 |
| **来电屏幕不亮** | PowerManager / full_screen_intent | PowerManagerService 看 `wakeUpInternal` |

## 六、面试高频考点

### Q1：来电流程中 RIL 和 Telecom 分别做了什么？

**RIL**：翻译 AT 命令，把 Modem 的物理层消息转成 Java 对象，通知 Phone 状态变化。

**Telecom**：管理通话状态机，处理多路通话，音频路由，通知 InCallUI 显示。

### Q2：为什么 CallTracker 要主动轮询（pollCallsWhenSafe），不直接等 Modem 推？

因为 Modem 推送的消息不完整。`RING` 只说有来电，`+CLIP` 只有号码。要拿到完整通话信息（通话 ID、状态、时长等），需要发 `AT+CLCC` 主动查询。

### Q3：Android 6.0 引入 Telecom 的目的是什么？

Android 6.0 引入 Telecom 之前，InCallUI 直接监听 Phone 状态，耦合严重。Telecom 引入后：
- 通话逻辑（Telecom）和 UI（InCallUI）解耦
- 第三方通话 App（如微信语音）可以通过 `ConnectionService` 接入系统通话管理
- 多路通话、蓝牙音频等统一管理

### Q4：如何区分来电是用户挂断还是对方挂断？

看 `DisconnectCause` 对象：
- `DisconnectCause.INCOMING_MISSED` → 未接听
- `DisconnectCause.INCOMING_REJECT` → 用户主动拒接（点了拒接按钮）
- `DisconnectCause.NORMAL` + Q.850 原因码 16 → 正常挂断（需结合方向判断是本地还是远端）
- 通过 `DisconnectCause.isIncomingCall()` 结合挂断方向判断

## 七、扩展知识

### 通话状态机

```
NEW → RINGING → ACTIVE → DISCONNECTED
                ↓
              HOLDING（来电等待）
                ↓
              TERMINATED（通话终止）
```

### ConnectionService

Android 6.0+ 第三方通话 App 接入系统的标准方式：
- 微信语音、WhatsApp 通话等通过 `ConnectionService` 注册
- 系统来电界面可以统一显示第三方来电
- `Telecom` 统一管理所有通话，不分来源
