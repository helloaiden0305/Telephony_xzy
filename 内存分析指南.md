## 一、Android 内存三剑客

如果把我们的手机系统比作一个庞大的"**科技园区**"，那么这三种内存就是园区里不同类型的"建筑空间"。

---

### 1. Heap 内存（Java 堆内存）—— "拎包入住的精装公寓"

#### 什么是 Heap 内存？

在 Android 开发中，Heap 内存通常指 **Java/Kotlin 堆内存（Dalvik/ART Heap）**。

这是 Android 系统专门为普通应用程序（App）分配的一块内存区域。你在代码里 `new` 出来的绝大多数对象（比如一个字符串、一个按钮、一个用户实体类），都存放在这里。

#### 核心特点

| 特点 | 说明 |
|------|------|
| **自带保洁阿姨（GC 机制）** | 你只需要负责"申请入住"（创建对象），不需要管"退房打扫"。系统有 GC（垃圾回收器）会定期巡逻，把没人用的对象清理掉 |
| **空间有限，容易"爆"** | 系统对每个 App 的 Heap 内存设置了严格的上限（如 256MB 或 512MB） |

#### 常见问题

```
一旦你的 App 里的垃圾对象太多，清理不过来，
或者塞进了一个超大物件，就会触发崩溃：

❌ OOM（Out Of Memory）
```

---

### 2. Native 内存（原生内存）—— "自由搭建的私人别墅"

#### 什么是 Native 内存？

Native 内存是分配给 **C/C++ 代码** 使用的内存空间。

在 Android 中，很多底层的框架（如图形渲染、音视频解码）以及使用了 NDK/JNI 技术的游戏引擎，都在这块内存里运行。

#### 核心特点

| 特点 | 说明 |
|------|------|
| **空间广阔** | 只要手机物理内存还有空余，它就能一直申请 |
| **没有保洁（手动管理）** | 开发者必须自己负责 `malloc`（申请）和 `free`（释放） |

#### 常见风险

```
⚠️ 内存泄漏（Native Leak）

如果你在 C++ 代码里建了一栋别墅，用完后忘了拆（忘记释放内存），
这块地皮就会一直被占用。

日积月累，整个手机的物理内存都会被耗尽，
最终导致系统为了自保，无情地把你的 App 甚至其他后台应用强杀掉
（Low Memory Killer）。
```

---

### 3. 共享内存（Shared Memory）—— "跨部门的公共会议室"

#### 什么是共享内存？

在 Android 系统中，每个 App 都是一个独立的进程，就像园区里不同公司的办公室，互相之间是隔离的（为了安全）。

但有时候，App 之间或者 App 与系统服务之间需要传递大量数据（比如把一张超高清图片传给屏幕渲染服务）。如果用常规方法复制一份传过去，太慢且太占内存。

这时候就需要 **共享内存（如 Ashmem、ION 机制）**。

#### 核心特点

| 特点 | 说明 |
|------|------|
| **极速沟通，零拷贝** | 系统在公共区域划出一块地，App A 把数据放进去，App B 直接去这块地里读 |
| **数据不需要复制两份** | 极大地节省了内存并提升了性能 |

#### 典型场景

- 流畅动画
- 跨进程传递超大 Bitmap 图片
- 相机预览画面的数据流转

---

### 💡 核心总结对比

| 内存类型 | 园区比喻 | 管理者 | 空间限制 | 常见问题 |
|---------|---------|--------|---------|---------|
| **Heap 内存** | 精装公寓 | 系统自动管理 (GC) | 有严格上限 (受限于虚拟机) | Java OOM (内存溢出) |
| **Native 内存** | 私人别墅 | 开发者手动管理 (C/C++) | 几乎无限制 (受限于物理内存) | Native 内存泄漏、被系统强杀 |
| **共享内存** | 公共会议室 | 系统底层映射 | 视系统分配而定 | 句柄泄漏、映射失败 |

## 二、LMK 内存查杀机制

### 1. LMK 是什么？它是"园区的终极保安"

LMK（Low Memory Killer）是 Android 系统底层的自我保护机制。它根本不关心你的 App 占用的是 Heap 内存（精装公寓）还是 Native 内存（私人别墅）。

**LMK 只看两个东西：**

| 判断依据 | 说明 |
|---------|------|
| **你的 App 总共占了多少物理地皮？** | 总 PSS/RSS 内存，包含 Java Heap + Native Heap + 共享内存等所有开销 |
| **你的 App 现在有多重要？** | 进程优先级/oom_adj，是正在前台被用户使用的 App，还是偷偷躲在后台的 App？ |

**工作机制：**

```
当整个手机的物理内存（RAM）快要耗尽时，
LMK 这个"保安"就会出动，
按照优先级把那些不重要的、占用内存大的后台 App 直接强杀（Kill）掉，
把地皮收回来给前台应用使用。
```

---

### 2. 为什么总把 Native 内存和 LMK 联系在一起？

既然 LMK 针对的是总内存，为什么在排查问题时，总觉得是 Native 内存惹的祸呢？

这是因为 **Heap 内存和 Native 内存的"死法"不同**：

#### Heap 内存（Java 堆）—— 通常死于"自杀"（OOM 崩溃）

```
系统给每个 App 的 Heap 内存设置了严格的上限（比如 512MB）。

如果你的 Java 代码疯狂创建对象，
还没等整个手机的内存用完，你的 App 就会先触碰到这 512MB 的天花板。

此时，App 会抛出 OutOfMemoryError（OOM）并自己崩溃闪退。

这属于"自杀"，根本轮不到 LMK 这个保安出手。
```

#### Native 内存 —— 通常死于"他杀"（被 LMK 强杀）

```
Native 内存没有这种单应用上限。

如果你的 C/C++ 代码发生了内存泄漏，它会一直申请内存：
1GB、2GB、3GB……它不会触发 Java 层的 OOM 崩溃。

但是，它这种无底洞式的索取，最终会把整个手机的物理内存（比如 8GB）全部吃光。

这时候，系统彻底没内存了，LMK 保安被迫出动，
发现你的 App 占了极大的空间，就会直接把你强杀掉。
```

---

### 💡 总结观点

```
Java 内存泄漏   →   触碰单应用上限   →   抛出异常，App 崩溃自杀（OOM）

Native 内存泄漏  →   无限制增长，耗尽手机总内存   →   触发系统保护，App 被系统强杀（LMK）
```

LMK 监控的是全局总内存，但因为 Native 内存没有上限限制，所以 **Native 内存泄漏往往是导致系统触发 LMK 疯狂杀后台的"罪魁祸首"**。

## 三、meminfo 日志解读

### 什么是 meminfo_all.txt？

它通常是通过在 adb 命令行中执行 `dumpsys meminfo` 命令抓取出来的。

详细记录了在抓取日志的这一瞬间，手机的内存（RAM）到底被谁占用了、占了多少、以及是怎么占用的。

---

### 1. 核心概念：RSS 与 PSS

| 指标 | 说明 | 准确性 |
|------|------|--------|
| **RSS** (Resident Set Size) | 进程实际占用的物理内存大小 | ❌ 不准，把"共享内存"也全算在自己头上了 |
| **PSS** (Proportional Set Size) | **最准确的内存指标**，把共享内存"AA 制"平摊了 | ✅ 排查内存泄漏通常看 PSS |

**举例：**

```
三个应用共享了 3MB 的基础库，

RSS 会给每个应用都算 3MB，

PSS 会给每个应用只算 1MB（AA 制平摊）。
```

---

### 2. 报告的四大核心板块

#### 板块一：按"进程"排名（Total PSS/RSS by process）

**意思：** 这是内存消耗的"排行榜"。

**示例解读：**

| 进程 | PSS | 说明 |
|------|-----|------|
| system | 177MB | 系统核心进程，占用大正常 |
| com.android.systemui | 135MB | 状态栏/导航栏等 UI，占用大正常 |
| com.hihonor.android.launcher | 118MB | 桌面，占用大正常 |

---

#### 板块二：按"重要程度"排名（Total PSS/RSS by OOM adjustment）

**意思：** Android 系统会根据进程的重要程度给它们分组，当内存不足时，会按照这个分组从下往上杀进程（OOM 机制）。

**分组解读：**

| 分组 | 说明 | 被杀优先级 |
|------|------|-----------|
| **Foreground** (前台进程) | 当前正在用的 | 绝对不能杀 |
| **Visible** (可见进程) | 用户能看到的 | 尽量不杀 |
| **Persistent** (常驻进程) | 系统核心服务 | 杀了手机就挂了 |
| **Cached** (缓存进程) | 后台缓存的应用 | **最底层的炮灰**，优先杀 |

**示例：**

```
Cached 占了 364MB，

当您打开新应用需要内存时，
系统会优先把这些后台缓存的应用（如 Chrome、应用市场等）
杀掉腾出空间。
```

---

#### 板块三：按"数据类型"排名（Total PSS/RSS by category）

**意思：** 这些内存到底装了些什么东西？

| 类型 | 说明 |
|------|------|
| **Native** | C/C++ 层代码分配的内存 |
| **Dalvik** | Java/Kotlin 代码分配的内存（通常说的堆内存） |
| **.so mmap / .apk mmap** | 加载的动态链接库和安装包占用的内存 |
| **GL mtrack / EGL mtrack** | 图形渲染（GPU）相关的内存 |

---

#### 板块四：全局大总结（最底部的几行，最重要！）

**这是整个手机内存的大盘数据：**

| 指标 | 示例值 | 说明 |
|------|--------|------|
| **Total RAM** | 3,893,596K | 手机的总物理内存，约 **4GB** |
| **status** | normal | 当前内存状态正常，没有处于极度紧缺状态 |
| **Free RAM** | 1,944,961K | 广义上的可用内存（约 1.9GB），包含 cached pss |
| **Used RAM** | 2,285,952K | 真正被死死占用的内存（约 2.2GB） |
| **ZRAM** | 281,624K physical used for 989,396K in swap | 内存压缩技术，约 989MB 不常用的数据被压缩成 281MB |

**ZRAM 说明：**

```
这是 Android 的内存压缩技术。

系统把大约 989MB 不常用的数据，
压缩成了 281MB 塞在物理内存里。

这相当于给 4G 的手机变相"扩容"了约 700MB。
```

---

### 💡 meminfo 总结

```
这份文件显示：
- 这是一台 4GB 内存的设备
- 当前系统运行状态良好（status normal），没有发生严重的内存溢出
- 系统 UI、桌面和 GMS 服务是当前的内存消耗大户
- ZRAM（内存压缩）正在积极工作，为系统节省了大量空间

排查建议：
如果您在排查卡顿或杀后台问题，
可以重点关注 Cached 列表里的应用是不是被频繁清理了。
```

## 四、快速总结

### 三剑客对比

| 内存类型 | 比喻 | 管理方式 | 上限 | 典型问题 |
|---------|------|---------|------|---------|
| **Heap** | 精装公寓 | GC 自动 | 有上限 | OOM 崩溃 |
| **Native** | 私人别墅 | 手动管理 | 无上限 | 泄漏→被 LMK 杀 |
| **共享** | 公共会议室 | 系统映射 | 系统分配 | 句柄泄漏 |

### LMK 查杀逻辑

```
Java 内存泄漏  →  OOM 自杀
Native 内存泄漏 →  LMK 他杀
```

### meminfo 关键指标

```
看总占用  →  PSS（准确）
看进程排名 →  找内存大户
看分类   →  定位泄漏类型
看 ZRAM  →  了解内存压缩情况
```
