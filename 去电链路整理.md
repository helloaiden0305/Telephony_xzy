# AOSP 去电链路：从拨号界面到 Modem 发起呼叫

> Android Telephony 去电流程，排查"拨不出去"必备

---

## 一、整体架构

去电是 **纯下行链路**，方向固定：

```
InCallUI (拨号盘) → Telecom → Telephony → RIL → RILD → Modem
```

和来电/挂断的区别：**去电是用户主动发起的，从一开始就走命令下发方向，不需要等 Modem 上报。**

---

## 一.5 Telephony vs Telecom 核心区分

整个链路跨越了两个层级，理解它们的职责差异对排查问题很重要：

| | Telephony | Telecom |
|--|--|--|
| **职责** | 管硬件，跟基站通信 | 管电话，管理通话状态和多路切换 |
| **核心类** | `RIL`, `GsmCdmaPhone`, `CallTracker` | `CallsManager`, `android.telecom.Call` |
| **去电中做什么** | 把号码发给 Modem（`ATD`） | 创建 OutgoingCall 对象，管理状态流转 |
| **桥梁** | **`ConnectionService`**（`TelephonyConnectionService`）负责把 Telephony 的底层通话包装成 Telecom 的统一通话对象 |

```
Dialer 拨号
  → Telecom（创建 OutgoingCall）
    → ConnectionService（桥梁，包装通话对象）
      → Telephony（执行 dial → RIL → ATD → Modem）
```

**Call 对象在去电流程中的角色变化：**

```
阶段 1：Telecom 发起 Call   →  CallsManager.startOutgoingCall() 创建 OutgoingCall（状态 NEW → CONNECTING）
阶段 2：Telephony 内部 Call  →  GsmCdmaCallTracker 执行 dial，底层 Call 承载 AT 命令结果
阶段 3：Telecom 对外 Call    →  状态流转 CONNECTING → DIALING → ALERTING → ACTIVE
阶段 4：UI 展示的 Call       →  InCallUI 绑定到 Telecom 的 Call，显示"通话中"
```

## 二、完整流程

```
用户在拨号盘输入号码 → 点击拨号按钮
  ↓
DialpadFragment → 拨号器拨号逻辑入口（具体类名因 Dialer 实现而异）.dial()
  ↓
TelecomManager.placeCall(Uri, Bundle)
  ↓
TelecomManager → TelecomService（跨进程调用）
  ↓
CallsManager.startOutgoingCall(phoneAccountHandle, address, extras)
  ↓
创建 OutgoingCall 对象，状态：NEW → CONNECTING
  ↓
ConnectionService → TelephonyConnectionService.onPlaceCall()
  ↓
GsmCdmaPhone.dial(number)
  ↓
GsmCdmaCallTracker.dial()
  ↓
RIL.java → mCi.dial() → RIL_DIAL()
  ↓
通过 socket 发送 RIL_REQUEST_DIAL 给 RILD
  ↓
vendor ril 库处理 RIL_REQUEST_DIAL
  ↓
发送 AT 命令：ATD13800138000;
  ↓
Modem 发起网络呼叫 → 返回 OK
  ↓
网络接通 → Modem 上报连接成功
  ↓
RIL → CallTracker → Telecom 状态：CONNECTING → ACTIVE
  ↓
InCallUI 显示通话中界面
```

## 四、关键类速查

| 类 | 层级 | 去电中的职责 |
|----|------|-------------|
| `DialpadFragment` | Dialer App | 拨号盘界面，用户输入号码 |
| 拨号器拨号入口 | Dialer App | 处理拨号按钮点击，调用 Telecom（具体类名因 Dialer 实现而异） |
| `TelecomManager` | framework/telecom | 对外 API，App 层拨号入口 |
| `CallsManager` | framework/telecom | 管理通话状态，创建 OutgoingCall |
| `TelephonyConnectionService` | framework/telephony | 桥接 Telecom 和 Telephony |
| `GsmCdmaPhone` | framework/telephony | Phone 对象，执行 dial |
| `GsmCdmaCallTracker` | framework/telephony | 管理通话状态，调用 RIL 发拨号请求 |
| `RIL.java` | framework/telephony | 封装 RIL_REQUEST_DIAL 发给 RILD |
| `rild` / vendor ril | vendor 层 | 收到 DIAL 请求，发 ATD 命令给 Modem |

## 五、拨号 AT 命令

| 命令 | 用途 | 示例 |
|------|------|------|
| `ATD<number>;` | 拨打语音电话 | `ATD13800138000;` |
| `ATD<number>;` + 特殊前缀 | 紧急号码 | `ATD110;` |
| `ATH` | 挂断已建立的通话 / 取消正在进行的拨号 | `ATH` |

**注意**：`ATD` 末尾必须有分号 `;`，这是 GSM 语音拨号的固定格式。

| 阶段 | RIL 日志 | Telecom 日志 |
|------|---------|-------------|
| 用户点击拨号 | — | 拨号器日志 `dial()`（具体类名因 Dialer 实现而异） |

| RIL 收到拨号请求 | `RIL_REQUEST_DIAL` | — |

| 网络接通 | 状态变化上报 | `CONNECTING → ACTIVE` |
| 拨号失败 | `RIL_E_DIAL_MODIFY_FAILED` | `call failed` |

## 七、排查思路（"为什么拨不出去"）

### 7.1 拨号请求根本没发出来

**排查方向**：从 UI 开始逐层检查

| 层级 | 查什么 | 日志关键词 |
|------|--------|-----------|
| Dialer App | 拨号按钮点击是否触发 | 拨号器日志 `dial()`（具体类名因实现而异） |
| TelecomManager | placeCall 是否调用 | `TelecomManager` `placeCall` |
| Telecom | CallsManager 是否收到请求 | `CallsManager` `startOutgoingCall` |
| ConnectionService | onPlaceCall 是否执行 | `TelephonyConnectionService` `onPlaceCall` |
| CallTracker | dial 是否被调用 | `GsmCdmaCallTracker` `dial` |

### 7.2 RIL 层没发出去

| 现象 | 原因 | 日志 |
|------|------|------|
| RIL 日志无 `RIL_REQUEST_DIAL` | 拨号请求没到 RIL 层 | logcat 搜 `RIL_REQUEST_DIAL` |
| RIL 报了 `RIL_E_INVALID_ARGUMENTS` | 号码为空或格式错误 | 检查拨号号码是否合法 |
| socket 发送超时 | RILD 进程无响应 | 看 RILD 进程是否存活 |

### 7.3 RILD 没发给 Modem

| 现象 | 原因 | 日志 |
|------|------|------|
| RILD 日志无 `ATD` | RILD 没收到或没解析 | RILD 层日志 |
| Modem 返回 `ERROR` | Modem 状态异常（如 SIM 未就绪） | 看 AT 通道返回 |
| AT 通道超时 | Modem 无响应 | 看 RILD 超时日志 |

### 7.4 拨号成功但网络未接通

| 现象 | 原因 | 排查 |
|------|------|------|
| ATD 返回 OK，但没接通 | 网络侧拒绝（欠费/无信号/黑名单） | 需要 rillog 看网络信令 |
| 立即返回 busy | 对方占线或网络拥塞 | 看 DisconnectCause |
| 拨号后几秒自动挂断 | 网络侧超时拒绝 | 看 Modem 侧信令日志 |

## 八、常见场景分析

### 场景一：点拨号没反应

```
排查步骤：
1. 拨号器拨号入口是否收到点击事件？ → 看拨号器 UI 日志
2. TelecomManager.placeCall 是否调用？ → 看 Telecom 日志
3. CallsManager 是否创建 OutgoingCall？ → 看 CallsManager
```

**大概率断在**：拨号号码为空，或 Telecom 未正确绑定。

### 场景二：拨号失败，提示"未连接网络"

```
排查步骤：
1. SIM 卡是否就绪？ → 看 Telephony 日志
2. 是否注册到网络？ → 看 Network 状态
3. RIL 是否发出 RIL_REQUEST_DIAL？ → 看 RIL 日志
4. Modem 返回什么？ → 看 AT 通道返回
```

**大概率断在**：SIM 未就绪或网络未注册。

### 场景三：拨号慢（点到接通 > 5 秒）

```
排查步骤：
1. UI 响应延迟？ → 看 InCallUI 日志
2. Telecom 处理慢？ → 看 startOutgoingCall 时间戳
3. Modem 响应慢？ → 看 ATD 到 OK 的时间差
4. 网络侧慢？ → 看 Modem 侧信令（需要 rillog）
```

**大概率断在**：网络侧处理慢（基站响应或核心网路由）。

## 九、与来电/挂断的对比

| 维度 | 去电 | 来电 | 挂断（用户主动） |
|------|------|------|-----------------|
| **方向** | 纯下行 | 纯上行 | 纯下行 |
| **触发** | 用户点击拨号 | Modem 上报 RING | 用户点击挂断 |
| **AT 命令** | `ATD<number>;` | 无（被动接收） | `ATH` |
| **RIL 请求** | `RIL_REQUEST_DIAL` | 无（Unsolicited） | `RIL_REQUEST_HANGUP` |
| **初始状态** | NEW → CONNECTING → DIALING → ALERTING → ACTIVE | NEW → RINGING | ACTIVE → DISCONNECTING |
| **最终状态** | CONNECTING → ACTIVE | RINGING → ACTIVE | ACTIVE → DISCONNECTED |

## 十、面试高频考点

### Q1：去电流程中 Telecom 做了什么？

收到 TelecomManager.placeCall 后，创建 OutgoingCall 对象，状态设为 CONNECTING，然后通过 ConnectionService 通知 Telephony 层执行拨号。完整状态流转为 `NEW → CONNECTING → DIALING → ALERTING → ACTIVE`。同时管理音频路由，确保拨号音正确输出。

### Q2：拨号后状态为什么是 CONNECTING 而不是直接 ACTIVE？

因为 `ATD` 命令发出后，Modem 需要等待网络侧响应（呼叫建立、对方振铃等）。CONNECTING 表示"已发起呼叫，等待网络接通"，这是一个中间状态，直到网络返回接通才变成 ACTIVE。

### Q3：如果拨号时 SIM 卡未就绪怎么办？

RIL 会返回 `RIL_E_SIM_NOT_READY` 或类似错误，Telephony 层会向 Telecom 报告拨号失败，InCallUI 显示"无法拨号"提示。不会往下发 ATD 命令。
